# Time Complexity

We will learn how to **calculate** the time complexity of a piece of code using Big O notation.

## Concept
Time complexity in Big O notation measures how an algorithm's running time increases with the size of its input.

### Statements run time
Each line of code takes a certain amount of time to run. We can estimate the time for each statement by first identifying its type.

1. Assignment, operations (+, -, *, etc.)

```java

int x = 1;  // 1
x = 1 + 2;  // 1 assignment + 1 addition(+)
x = y;      // 1
```
The time complexity for the above code is `O(1)`.

2. Comparison

```java
if (condition){}  // 1 comparison
```

The time complexity for the above code is `O(1)`.

3. Return

```java
return result;    // 1
```

The time complexity for the above code is `O(1)`.

4. Loop
A loop's time depends on how many iterations it performs.
The following code takes `O(n)` time.

```java
for (i = 1; i <= n; i++){} // 1 assignment, (1 comparison + 1 increment) done n times
```

- The loop iterates until `i` reaches `n`, since the comparison is `(i <= n)`.
- `(i = 1)` → 1 assignment.
- `(i <= n)` → 1 comparison, performed `n` times (once per iteration).
- `(i++)` → 1 increment, performed `n` times (once per iteration).
- Time calculation for the above code:
  - T(1 + (1 + 1) * n)
  -  T(1 + 2n)
  -  Ignore constants (1, 2)
  -  T(n)
  -  `O(n)`

>[!Important]
> *Ignore constants* is one of the Big O rules.

### Big O rules and examples

Big O follows the rules listed below when calculating time complexity.

1. **Ignore constants**

```java
int x = 0;                     // Line #1
for (i = 1; i <= n; i++){      // Line #2
    x = i + 1;                 // Line #3
}
```
- Time analysis:
```
Line #1: 1 assignment
Line #2: 1 assignment, (1 comparison + 1 increment) done n times  ( 1 + (1+1)*n )
Line #3: (1 assignment + 1 addition) done n times                 ( (1+1)*n )
```

- Time calculation:
  - T(1 + (1 + 1)*n + (1 + 1)*n)
  - T(1 + 2n + 2n)
  - T(1 + 4n)
  - Ignore constants (1, 4)
  - T(n)
  - `O(n)`

2. **Drop lower-order terms**

```java
for (i = 1; i <= n; i++){       // Line #1
    for (j = 1; j <= n; j++){   // Line #2
        x = y + z;              // Line #3
    }
}
```
- Time analysis:
```
Line #1: 1 assignment, (1 comparison + 1 increment) done n times                  ( 1 + (1+1)*n )
Line #2: 1 assignment done n times, (1 comparison + 1 increment) done n*n times   ( 1*n + (1+1)*n*n )
Line #3: (1 assignment + 1 addition) done n*n times                               ( (1+1)*n*n )
```

- Time calculation:
  - T(1 + 2n + n + 2n² + 2n²)
  - T(1 + 3n + 4n²)
  - Drop lower-order terms (3n), ignore constants (1, 4)
  - T(n²)
  - `O(n²)`

3. **Combine variables only if they refer to the same input**

```java
int a = 0;
int b = 0;

for (i = 1; i <= n; i++){       // Line #1
    a = a + 1;                  // Line #2
}

for (i = 1; i <= m; i++){       // Line #3
    b = b + 1;                  // Line #4
}
```
- Time analysis:
```
Line #1: 1 assignment, (1 comparison + 1 increment) done n times  ( 1 + (1+1)*n )
Line #2: (1 assignment + 1 addition) done n times                 ( 2n )
Line #3: 1 assignment, (1 comparison + 1 increment) done m times  ( 1 + (1+1)*m )
Line #4: (1 assignment + 1 addition) done m times                 ( 2m )
```

- Time calculation:
  - T(1 + 2n + 2n + 1 + 2m + 2m)
  - Combine variables that share the same input: (2n, 2n), (2m, 2m)
  - T(2 + 4n + 4m)
  - Ignore constants (2, 4, 4)
  - T(n + m)
  - `O(n + m)`

>[!NOTE]
> `n` and `m` are kept separate because they represent two *different, independent* inputs; you can't assume one is larger than the other, so they can't be combined or dropped the way same-input terms can.

4. **Always take the worst-case time complexity**

Consider the pseudocode below:

```java
if (condition) {
    for (i = 1; i <= n; i++) {   // Line #1
        // ...
    }
} else {
    x = 8;                        // Line #2
}
```

- Time analysis:
```
Line #1: 1 assignment, (1 comparison + 1 increment) done n times  (1 + (1+1)*n)  -> O(n)
Line #2: 1 assignment                                             (1)            -> O(1)
```

The time complexity of the code above is `O(n)`, since we always take the worst case, and `O(n)` > `O(1)`.


## Example
In this section, we'll take a piece of code and improve its time complexity.

**Linear Search** checks every element one by one until it finds the target (or reaches the end):

```java
public static int linearSearch(int[] arr, int target) {
    for (int i = 0; i < arr.length; i++) {   // {( n )}
        if (arr[i] == target) {               // {( n )}
            return i;                          // {( 1 )}
        }
    }
    return -1;                                 // {( 1 )}
}
```

- Time calculation:
  - T(1 + 2n + 1)
  - T(2 + 2n)
  - Ignore constants
  - T(n)
  - `O(n)` (worst case: target is last, or not present at all)

This works on *any* array, sorted or not. But if we know the array is **sorted**, we can do much better with **Binary Search**: check the middle element, and depending on whether the target is smaller or larger, recurse into *only* the half that could still contain it while completely discarding the other half.

```java
public static int binarySearch(int[] arr, int low, int high, int target) {
    if (low > high) {                                           // {( 1 )}
        return -1;                                              // {( 1 )}
    }

    int mid = low + (high - low) / 2;                           // {( 1 )}
    if (arr[mid] == target) {                                   // {( 1 )}
        return mid;                                             // {( 1 )}
    } else if (arr[mid] < target) {
        return binarySearch(arr, mid + 1, high, target);        // {( T(n/2) )}
    } else {
        return binarySearch(arr, low, mid - 1, target);          // {( T(n/2) )}
    }
}
```

- Time calculation:
  - T(n) = T(n/2) + O(1)
  Notice that exactly one branch executes and the other half is thrown away entirely, producing: T(n) = `O(log n)`.

Let's visualize the process: if the array has `n` elements, the size is cut in half on every recursive call:
 
```
n
n/2
n/4
n/8
n/16
...
1
```
 
This halving pattern is exactly what `log n` measures: an array of 8 elements takes 3 halving steps to shrink down to a single element, which is the same as log₂(8) = 3, matching the O(log n) time complexity we derived above. 
In general, an array of `n` elements takes log₂(n) halving steps, which is why binary search's recursion depth (and total time) is `O(log n)`.
